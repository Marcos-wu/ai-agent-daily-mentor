# 每日任务模板

本文件用于生成适合 Python 零基础或基础薄弱学习者的每日任务。目标是让学习者能照着做、能跑通、能理解为什么这么做，而不是把知识点压缩成摘要。

## 适用场景

- 用户要求“生成 Day N”。
- 用户要求“每日任务 / 今天任务 / 明天任务”。
- 用户要求“零基础版 / 保姆级讲解 / 新手友好感”。
- 用户要求把课程内容保存为英文目录下的 Markdown 文件。
- 用户反馈概念没懂、代码没跑通、需要从零讲起。

## 输出结构

每日任务按以下结构输出：

- `# 📅 Day N：主题`
- `## 🧭 今日方向`
- `## 🧠 先用生活比喻搞懂`
- `## 🎯 今天只做这三件事`
- `## 🪜 手把手路线（总共 X 小时）`
- `## 💻 跟写代码区`
- `## 🚑 卡点急救包`
- `## 🧩 概念对照表`
- `## ✅ 今日验收清单`
- `## 📝 今日复盘小纸条`
- `## 📥 明日同步接口`

## 内容颗粒度

- 每天只讲 2-3 个核心任务。
- 每个抽象概念尽量配生活比喻。
- 每一步都要说明“做什么、为什么做、看到什么算成功、失败时怎么排查”。
- 代码尽量短，但必须完整可运行。
- 不把高级概念塞进每日任务；如果必须提到，只用一句话解释，不展开。
- 默认按零基础可承受的 4-5 小时设计；用户明确要求时再扩展到 6 小时。

## 文件输出规则

目录名统一推荐英文：

```text
weekly-outlines/
daily-tasks/
knowledge-notes/
```

文件名可以中英文混合，推荐：

```text
weekly-outlines/Week1-Python-MUP-HTTP-Git.md
daily-tasks/Day1-Python工程环境与函数基础.md
daily-tasks/Day2-HTTP与API调用.md
knowledge-notes/Python虚拟环境.md
```

如果用户提供了已有目录，优先遵从用户路径；如果没有给目录，只输出建议路径和 Markdown 内容。

## 双链策略

每日任务使用克制双链：

- 标题或首段中给核心概念加双链。
- `关联知识：` 中列出 3-5 个稳定双链。
- 验收题或验收清单中可以少量加双链。
- 代码块内部禁止双链。
- 文件名、命令、环境变量、API 路径用反引号，不加双链。

推荐优先使用这些稳定标题：

- `[[Python 虚拟环境]]`
- `[[环境变量与 API Key]]`
- `[[.gitignore 配置]]`
- `[[JSON 文件读写]]`
- `[[Python 函数]]`
- `[[异常处理]]`
- `[[HTTP 请求结构]]`
- `[[HTTP 状态码]]`
- `[[OpenAI-compatible API]]`
- `[[LLM messages 结构]]`
- `[[LLMClient 封装]]`
- `[[对话历史持久化]]`

## Day 1-Day 3 风格样例

- Day 1：用“私人厨房”解释 `[[Python 虚拟环境]]`，用“保险箱”解释 `[[环境变量与 API Key]]`，用“拒收清单”解释 `[[.gitignore 配置]]`。
- Day 2：用“快递系统”解释 `[[HTTP 请求结构]]`，用“点餐窗口”解释 API，用“签收状态”解释 `[[HTTP 状态码]]`。
- Day 3：用“电话客服中心”解释 `[[OpenAI-compatible API]]`，用“微信群里的三个人”解释 `[[LLM messages 结构]]`，用“随机程度旋钮”解释 temperature。

## 每周大纲生成规则

每周大纲也必须服务于每日任务：

- 写清本周目标。
- 写清本周最低可交付成果。
- Day 1-Day 7 每天都要有一个可执行任务。
- 每天写一个卡点预警。
- 周末必须安排复盘、整理 README 或项目表达。
- 总周期固定为 8 周，不新增第 9 周。

## 质量检查

生成后自查：

- 是否使用“每日任务”命名。
- 是否没有生成标准版内容。
- 是否使用英文目录名。
- 是否只聚焦三件事以内。
- 是否每一步都有成功检查点。
- 是否有卡点急救包和验收清单。
- 代码块内部是否没有 `[[...]]`。
- 双链是否克制且使用稳定命名。
