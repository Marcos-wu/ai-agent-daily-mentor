# Obsidian 知识点笔记维护规范

本文件用于解决“双链存在，但链接目标为空”的问题。默认策略是轻量化：知识点笔记只承载相关知识，不承担完整课程讲义。

## 目录

- 默认轻量笔记结构
- Python 基础详细度评分规则
- 生成级别
- Python 基础详细笔记结构
- 优先加深的 Python 基础知识
- Day 1 默认优先补充对象
- 从已有 Day 笔记反向生成笔记
- 增量补充规则
- 推荐输出格式

## 默认轻量笔记结构

普通知识点使用以下结构：

```markdown
# 知识点名称

## 核心解释

## 使用场景

## 常见坑

## 相关链接
```

默认长度：

- `核心解释`：3-6 句话。
- `使用场景`：2-4 条。
- `常见坑`：2-4 条。
- `相关链接`：3-6 个双链。

禁止默认生成复杂结构，例如：

- 来源记录。
- 今日新增。
- 完整教程。
- 长篇面试稿。
- 过长的历史演化。

## Python 基础详细度评分规则

Python 基础知识可以比其他知识点更详细。详细程度由“出现次数 + 出现位置 + 求职重要性”共同决定。

### 出现次数

- 当天出现 1 次：1 分。
- 当天出现 2-3 次：2 分。
- 当天出现 4 次以上：3 分。

### 出现位置

- 出现在 `📌 今日核心死磕` 或 `💻 今日代码实战`：3 分。
- 出现在 `🎯 6小时精细化路线` 或 `🎯 每日自我验收题`：2 分。
- 只出现在关联链接或相关链接中：1 分。

### 求职重要性

- Python 基础高频面试/项目必用：3 分。
- 工程辅助能力：2 分。
- 背景知识：1 分。

## 生成级别

- 3-4 分：轻量笔记。
- 5-7 分：标准笔记。
- 8 分以上：Python 基础详细笔记。

## Python 基础详细笔记结构

高分 Python 基础知识使用以下结构，仍保持克制，不写成长篇教程：

```markdown
# 知识点名称

## 核心解释

## 为什么它是 Python MUP

## 在 Agent 项目中的使用场景

## 最小代码例子

## 常见坑

## 相关链接
```

要求：

- `核心解释`：5-8 句话。
- `为什么它是 Python MUP`：说明它为什么是最少必要知识。
- `在 Agent 项目中的使用场景`：绑定 LLM CLI、FastAPI、RAG 或 AI-Interview。
- `最小代码例子`：只放一个短例子，必须可运行。
- `常见坑`：3-5 条。
- `相关链接`：3-8 个稳定双链。

## Day 1 默认优先补充对象

基于 Day 1：Python 工程环境与函数基础，优先补以下 5 个：

- `[[Python 虚拟环境]]`：详细。
- `[[环境变量与 API Key]]`：详细。
- `[[JSON 文件读写]]`：详细。
- `[[Python 函数]]`：标准。
- `[[异常处理]]`：标准。

不需要 Day 1 一次性补全所有双链，例如 `[[.gitignore 配置]]` 可以先保持轻量或暂不补。

## 从已有 Day 笔记反向生成笔记

当用户提供已有 Day 笔记路径时，执行以下逻辑：

1. 提取所有 `[[...]]` 双链。
2. 统计每个双链出现次数。
3. 判断其出现位置：核心死磕、代码实战、路线、验收题、普通关联。
4. 按评分选出 3-5 个最值得补的知识点。
5. 为这些知识点生成轻量、标准或详细笔记。
6. 不修改 Day 主笔记结构。

如果 agent 有文件写入权限并且用户要求写文件，可创建或更新对应 Markdown 文件；如果只是聊天输出，则输出可复制的 Markdown 文件块。

## 增量补充规则

同一知识点后续再次出现时：

- 不新建同义词笔记。
- 不整篇重写已有笔记。
- 只补充缺失内容或新增 1-3 条。
- 如果原笔记明显错误，可以标注“修正”并替换错误段落。

## 推荐输出格式

当需要输出知识点笔记时，使用文件块：

```markdown
文件：知识卡片/Python 虚拟环境.md

# Python 虚拟环境

## 核心解释

...
```

每天最多建议生成 3-5 个文件块。
