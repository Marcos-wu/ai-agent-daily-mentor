# 每日反馈与周计划自适应规则

生成下一天内容前，先根据用户反馈判断学习状态。不要机械推进周大纲。

## 状态分类

### 正常推进

判断条件：

- 当天核心代码能运行。
- 用户能完成主要验收标准。
- 没有阻塞性 Bug。

处理方式：

- 明天按当前周大纲继续。
- 可以在每日内容中保留少量复习，但不改变主线。

### 放慢复习

判断条件：

- 核心代码没有跑通。
- 环境没有配好。
- 用户明确说概念没懂。
- 出现阻塞性报错，且用户无法定位。

处理方式：

- 下一天降低 30%-50% 难度。
- 安排“复盘 + 最小修复 + 小步重写”。
- 不引入大型新框架或复杂新概念。
- 保留本周核心产出物，但缩小当天目标。

### 加难扩展

判断条件：

- 用户提前完成。
- 用户能讲清核心概念。
- 代码能运行，并完成扩展任务。

处理方式：

- 主线不跳跃。
- 给明天增加一个工程化扩展：日志、测试、README、异常类、`.env.example`、Git 分支或代码重构。

### 重排本周

判断条件：

- 连续两天卡住。
- Day 1 环境失败。
- 本周核心产出物已经明显无法按原计划完成。

处理方式：

- 重排剩余 Day 内容。
- 缩小本周交付目标，但不取消项目产出。
- 优先保证能跑、能讲、能 commit。

### 求职项目完整度不足

判断条件：

- 用户只完成了单点 Demo，没有后端服务化。
- 项目没有数据库持久化、缓存/异步任务、README、Docker 或测试。
- 项目只有单独 RAG 或单独 Agent，没有业务闭环。
- 用户不会解释项目亮点、业务流程和工程化价值。

处理方式：

- 下一天优先补工程化，不继续堆新技术。
- 如果是 RAG Demo，补 FastAPI API、题库数据结构、检索日志和引用来源。
- 如果是 Agent Demo，补工具调用记录、业务状态、失败处理和报告输出。
- 如果不会讲项目，安排项目表达训练：项目简介、技术实现、为什么这么设计、遇到的问题、如何优化。
- 如果目标是尽快投递，优先补 GitHub、README、演示链接、简历话术和个人优势。

## 内部判断清单

每日教学模式开始前，先内部判断：

- 当前周大纲是否存在；不存在则先生成或重建。
- 用户昨日反馈属于哪种状态。
- 今日安排是否仍服务于本周核心产出物。
- 今日是否需要补救、正常推进、扩展或重排。
- 今日是否需要补求职项目完整度：业务闭环、工程化、项目表达或投递材料。

如果需要调整，在 `📌 今日核心死磕` 开头用 1-2 句说明，例如：

“根据你昨天反馈 JSON 读写还没跑通，今天先不推进 HTTP API，改为把配置读取和历史保存彻底打牢。”
